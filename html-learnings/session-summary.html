1

Same Embedding Model for Indexing and Querying

"Cand faci extragerea din RAG trebuie sa folosesti acelasi model cu care ai facut initial embedingul?"
Yes. Each embedding model creates its own unique vector space. Model A encodes "contract termination" as [0.12, -0.87, ...] while Model B encodes it as [0.95, 0.11, ...]. Cosine similarity between vectors from different models is meaningless — like comparing coordinates from two different map projections.

If you switch models, you must re-embed ALL documents.

2

How HNSW Works

"How does HNSW work? What does it actually do?"
HNSW = Hierarchical Navigable Small World. An approximate nearest neighbor (ANN) index.

Structure: Multi-layer graph pyramid. Layer 0 has ALL vectors. Each upper layer is a sparser random subset with longer-range connections.

Search: Start at top layer (few nodes, big jumps) -> greedy hop to the closest node -> drop one layer -> refine -> repeat until Layer 0 -> return top-k results.

Trade-off: ~95-99% recall (might miss the absolute best match) but 100-200x faster than brute force.

Without HNSW: Seq Scan ~450ms. With HNSW: Index Scan ~2ms.

3

The Three HNSW Parameters: m, ef_construction, ef_search

"Can you explain ef_construction and ef_search? I'm not sure I understand the difference."
m = max edges per node when inserted. More edges = better recall, more RAM.

ef_construction = candidate list size during INDEX BUILD. Controls how carefully neighbors are chosen. Paid once. Can't change without rebuilding. Higher = better graph quality but slower build.

ef_search = candidate list size during QUERY. Controls how thoroughly the graph is explored per query. Tunable at runtime (SET hnsw.ef_search = 100). Higher = better recall but slower queries.

Key insight: ef_construction > m because the candidate list is the "interview pool" — you interview ef_construction candidates to pick the best m. A larger pool finds better neighbors.

Why they matter: The algorithm only explores neighbors of nodes that are IN the candidate list. If a node is rejected (list full, too far), its neighbors are NEVER explored. A small ef means entire graph branches become unreachable.

Layer 0 = All Chunks in the RAG Database

"On Layer 0 is the actual total chunks we have in the RAG DB?"
Yes. Layer 0 = ALL vectors. Upper layers are random subsets of the SAME nodes.

Example with 100,000 chunks: Layer 0 = 100,000 nodes. Layer 1 ~ 8,700. Layer 2 ~ 750.

Upper layers use the same node IDs as Layer 0 — they're not different data. When you find the best node in Layer 2, you drop to that same node in Layer 1.

5

Why ef_construction Needs to Be Higher Than m

"If m is 2, why don't we set ef_construction to 2 as well?"
Analogy: hiring. m = how many you hire. ef_construction = how many you interview. Interview 2 people for 2 positions? You'll miss great candidates.

Technical reason: The graph has branches. With ef=2, the candidate list fills up fast. When a "gateway" node is rejected, its entire branch (which might contain the BEST neighbors) is never explored. With ef=4+, the gateway stays in the list long enough for its neighbors to be discovered.

6

Operator Class Must Match Query Operator

"Can you explain vector_cosine_ops and the <=> operator better?"
The index is built for ONE specific distance metric. The query must use the matching operator.

vector_cosine_ops + <=> = cosine distance (used for text embeddings)
vector_l2_ops + <-> = Euclidean distance
vector_ip_ops + <#> = inner product

If they don't match, PostgreSQL ignores the index and falls back to sequential scan. Like organizing a library by topic but searching by page count — useless.

7

ONE Graph for ALL Chunks, Not One Per Chunk

"Why do we only have one HNSW graph? I thought each chunk gets its own small graph."
There is ONE graph. Each chunk embedding is ONE node. Edges connect nodes with similar embeddings.

When you INSERT a new chunk, PostgreSQL automatically adds it as a new node and connects it to its m nearest existing nodes. The graph grows but stays unified.

Nodes = rows in document_chunks table (each chunk's embedding)
Edges = "these two chunks have similar embeddings" (cosine proximity)

The SQL Query Explained

"In ORDER BY embedding <=> '[0.12, 0.45, ...]'::vector — embedding is the column and [0.12, ...] is the query vector?"
Yes.
embedding = the column (each row's chunk vector)
'[0.12, ...]'::vector = the query vector (user's question, embedded with same model)
<=> = cosine distance operator

The HNSW index traverses the graph instead of comparing every row.

9

HNSW Index Auto-Updates on INSERT

"When you add a new document, is the HNSW run automatically so the index graph is updated?"
Yes. Just like a B-tree index, PostgreSQL automatically updates the HNSW graph on every INSERT. No manual rebuild needed.

Manual rebuild only for: first-time creation, changing operator class, changing m or ef_construction, or optional REINDEX after many deletes.

HNSW vs IVFFlat

"What's the difference between HNSW and IVFFlat?"
HNSW = graph-based. One connected graph. Traverse edges greedily. Fast (~2ms), high RAM.
IVFFlat = cluster-based. Vectors grouped into clusters by centroids. Find nearest centroid, scan that cluster. Less RAM, slower, can miss results at boundaries.
