Two Separate Pipelines

Lectia 3 and 4 teach two distinct pipelines that solve different problems but complement each other.

Pipeline 1: Structured Extraction (Lectia 3)

Turn messy documents into clean, typed JSON using an LLM + Pydantic schemas.

  • Input: PDF, DOCX, TXT, CSV
  • Output: JSON files (factura.json, contract.json)
  • Uses: Gemini + Pydantic response_schema

Pipeline 2: RAG with Embeddings (Lectia 4)

Enable semantic search over document content using vector embeddings + pgvector.

  • Input: PDF, DOCX, TXT
  • Output: Chunks + embeddings in PostgreSQL
  • Uses: SentenceTransformers + pgvector + HNSW

Key difference at Step 3

Extraction: LLM reads the text and fills in a Pydantic schema (structured fields).
RAG: Embedding model converts text into a vector (semantic meaning). No LLM involved at indexing time.

Extraction Pipeline -- Why Chunk Before Extract?

The LLM does the extraction. Chunking prepares the text so the LLM can handle it.

LLM context window limit

A 50,000-char report can exceed the LLM's input limit. Chunks keep it within bounds. Short documents (invoices ~2000 chars) skip chunking entirely.

Better extraction quality

Smaller, focused text = more precise extraction. The LLM doesn't get lost in irrelevant sections when extracting specific fields.

Can't split after extracting

After extraction you have a Pydantic object (Python object with fields). You can only split text. So chunking must come first.

The slide says "Text splitters (daca e mare)" -- chunking is conditional. See slide43.py:

def should_chunk(document, max_size=4000): return len(document.page_content) > max_size # Invoice (~2,000 chars) -> process directly, no chunking # Contract (~8,000 chars) -> chunk # Report (50,000+ chars) -> chunk

RAG Pipeline -- Why Chunk Before Embed?

Similar reasons, but for embeddings instead of LLM extraction.

Embedding models have token limits

MiniLM: 512 tokens max. OpenAI: ~8k tokens. A full PDF exceeds this -- text gets truncated and you lose information.

One embedding = one meaning

Embedding an entire contract compresses penalties + payments + confidentiality into one blurred vector. Chunking keeps each embedding focused on one topic.

Retrieval returns chunks, not docs

RAG injects top-k chunks into the LLM prompt. Whole documents would overflow the context window and waste tokens on irrelevant sections.

Where the extracted data lives in the database

The extracted JSON goes into the documents.metadata column (JSONB type in PostgreSQL, from slide13.py). This makes it queryable with SQL: WHERE metadata->>'prestator' = 'TechDev'. The RAG chunks go into the separate document_chunks table with the embedding vector column.

The tool definition is what the LLM sees

The Field(description=...) on each parameter tells the LLM what to fill in. The same Pydantic pattern from Lectia 3 (extraction schemas) is reused for tool parameters. The LLM's only job is understanding user intent and mapping it to parameters.

Hybrid Search: JSONB Filter + Vector Search

The correct way: filter FIRST in SQL, THEN search only within filtered documents.

WRONG: Filter after search

Search all chunks globally (top_k=5), then filter by doc_ids. Problem: relevant TechDev chunks might not make it into the global top 5.

CORRECT: Filter before search

Add WHERE document_id IN (...) before the HNSW traversal. PostgreSQL searches only within TechDev's chunks. Top 5 are all relevant.

def similarity_search(self, query_embedding, top_k=5, doc_ids=None): similarity = (1 - DocumentChunk.embedding.cosine_distance(query_embedding)).label("score") stmt = ( select(DocumentChunk, similarity) .options(joinedload(DocumentChunk.document)) .order_by(DocumentChunk.embedding.cosine_distance(query_embedding)) .limit(top_k) ) # Filter BEFORE vector search if doc_ids is not None: stmt = stmt.where(DocumentChunk.document_id.in_(doc_ids)) rows = self.db.execute(stmt).all() return [(chunk, float(score)) for chunk, score in rows]

The JSONB filter uses slide22-2.py filter_by_metadata(key, value). The similarity search uses slide57-2.py ChunkRepository.

How the pass from JSONB filter to vector search actually works

The course teaches the two repositories separately (slide22-2.py and slide57-2.py) but leaves the wiring as an exercise. The tool function is where you connect them. Here's the full chain:

The existing course code (3 separate pieces)

DocumentRepository (slide22-2.py) -- queries the documents table JSONB column. Returns Document objects.
ChunkRepository (slide57-2.py) -- queries the document_chunks table with vector search. Returns chunks + scores.
RAGService (slide58.py) -- wraps ChunkRepository, handles embedding the query string.

These exist independently. The course doesn't show them wired together -- that's the glue you'd write in the tool function.

# ────────────────────────────────────────────────────── # THE TOOL FUNCTION -- the glue that connects them # ────────────────────────────────────────────────────── @register_tool def search_documents(params: SearchDocumentsParams) -> str: """Cauta documente folosind search semantic + filtre structurate.""" with transaction() as db: doc_ids = None # STEP 1: JSONB filter (slide22-2.py) if params.prestator: doc_repo = DocumentRepository(db) docs = doc_repo.filter_by_metadata("prestator", params.prestator) # docs = [Document(id=3, ...), Document(id=7, ...)] doc_ids = [d.id for d in docs] # doc_ids = [3, 7] <-- just a list of integers # STEP 2: pass doc_ids into semantic search (slide57-2.py + slide58.py) rag = RAGService(db) results = rag.search(params.query, top_k=5, doc_ids=doc_ids) # internally: WHERE document_id IN (3, 7) ORDER BY embedding <=> query_emb return "\n\n".join( f"[{chunk.document.filename}] {chunk.content}" for chunk, score in results )

The pass is just a list of integers

Step 1: DocumentRepository.filter_by_metadata() -- queries documents table
SELECT * FROM documents WHERE metadata->>'prestator' = 'SC TechDev SRL'
Returns: [Document(id=3, filename="techdev_contract.txt"), Document(id=7, filename="techdev_addendum.txt")]
v
The glue -- one line of Python
doc_ids = [d.id for d in docs]
Result: [3, 7]
v
Step 2: ChunkRepository.similarity_search() -- queries document_chunks table
SELECT * FROM document_chunks WHERE document_id IN (3, 7) ORDER BY embedding <=> query_emb LIMIT 5
Returns: top 5 most similar chunks, but ONLY from TechDev's documents
Two tables, two repositories, one list of IDs connecting them

filter_by_metadata queries the documents table (where the extracted JSONB metadata lives). similarity_search queries the document_chunks table (where the RAG embeddings live). The connection is doc_ids = [3, 7] -- a plain Python list passed from one query to the next. That's all the "hybrid" is: one SQL query feeding IDs into another.

Why module-level _process_one?

Python's multiprocessing uses pickle to send functions to worker processes. Module-level functions are picklable; class methods are not. That's why _process_one is outside the class.

Why new pipeline per worker?

The Gemini API client (genai.Client()) isn't picklable. Each worker creates its own client. The @property with lazy init ensures this happens only once per process.

Continue-on-error

Try/except wraps each file. A corrupt PDF doesn't kill the batch -- it goes into results["errors"] and processing continues for the rest.

as_completed() order

Results arrive in the order files finish, not the order they were submitted. A fast 1-page invoice returns before a slow 50-page contract.

The schema parameter makes it generic

schema: Type[BaseModel] accepts any Pydantic class. Same batch_extract() works for Invoice, Contract, or any new document type -- just pass a different schema.

Cloud Architecture: S3, Watcher & Pipeline

How the pipeline runs in production (from Lectia 3, ~min 49-59). The pipeline and the chat app are separate systems communicating through a shared storage location.

Input
S3 Bucket -- Raw Documents
New PDFs, DOCX, TXT uploaded here
v
Runs separately (Lambda, job, cron)
Processing Pipeline
Load -> Chunk -> Extract/Embed -> Save output
v
Output
S3 Bucket -- Processed Data
JSON files, chunks, labels, embeddings
v
Detects new processed files
Watcher / Trigger / Cron Script
Watches the OUTPUT bucket, not the raw input
v
Loads new data
PostgreSQL + pgvector
New chunks + embeddings inserted, HNSW auto-updates
v
Uses updated data
Chat App / RAG Agent
Semantic search on fresh data, answers with citations

Watcher watches OUTPUT, not INPUT

The watcher in the chat app monitors the location where the pipeline saves its results (processed data), not where raw documents are uploaded. It detects that processing is done and loads the results.

Two update strategies

  • Automatic: Watcher/trigger detects new files -> loads into DB immediately
  • Scheduled: Cron script runs once/day, reads new output, inserts into DB

Pipeline runs "in alta parte"

The pipeline and the chat app are decoupled. Pipeline can be a Lambda function, a batch job, a separate server. They communicate only through the shared S3 location.

HNSW auto-updates on INSERT

When new chunks are inserted into PostgreSQL, the HNSW index updates automatically. No manual rebuild needed. The chat app immediately sees the new data.

Key idea from the lecturer

The pipeline and app don't need to know about each other. They just need to agree on a shared storage location (S3 bucket). The pipeline writes there, the app reads from there. This is a simple, decoupled architecture that works at any scale.

How Extracted Data Enters PostgreSQL

Nothing is automatic. Three separate steps, done at different times, by different people/tools.

Step 1: Define the table (once, by developer)

You write a SQLAlchemy model with a JSONB column. The column has no fixed shape -- it accepts any valid JSON. You don't define prestator, total, etc. at the database level.

slide13.py

Step 2: Create the table (once, by Alembic)

Run alembic upgrade head to execute the migration. This creates the actual table in PostgreSQL. The JSONB column exists but is empty.

slide55.py

Step 3: Fill it at runtime (every document)

Your extraction code calls repo.create(**extraction.to_db_format()). This INSERT puts the extracted fields into the JSONB column.

Why JSONB is powerful here

The JSONB column is schema-less. An invoice row can have {"numar", "total", "client"} while a contract row has {"prestator", "beneficiar", "valoare"} -- in the same column, in the same table. No migration needed when you add a new document type or extract a new field. PostgreSQL accepts any valid JSON and lets you query it with ->> operators.

Chunking: Two Different Roles in Each Pipeline

The same word "chunking" means different things depending on which pipeline you're in.

In the Extraction Pipeline

Purpose: Make text small enough for the LLM's context window.

When: Only if document > ~4000 chars. Short invoices skip it.

After chunking: Join first 3 chunks back into one text, one LLM call, one Pydantic result.

Are chunks saved? No. Chunks are temporary -- only the extracted fields (numar, total, prestator) survive into the database.

In the RAG Pipeline

Purpose: Create focused text pieces that each get their own embedding.

When: Always, for every document.

After chunking: Each chunk gets embedded separately, stored as its own row.

Are chunks saved? Yes. Every chunk is a row in document_chunks with its embedding.

Same file, two pipelines, different outcomes:

factura_001.txt (~800 chars, short document) Extraction Pipeline: load -> skip chunking (< 4000 chars) -> Gemini extracts -> save to documents.metadata Result: 1 row in documents table (JSONB: numar, total, client...) Chunks saved? No RAG Pipeline: load -> chunk always (500 chars each) -> embed each chunk -> save to document_chunks Result: 2 rows in document_chunks (each with embedding vector) Chunks saved? Yes, permanently
contract_servicii.txt (~3500 chars, larger document) Extraction Pipeline: load -> skip chunking (< 4000 chars, still fits) -> Gemini extracts -> save to documents.metadata Result: 1 row in documents table (JSONB: prestator, beneficiar, valoare...) Chunks saved? No RAG Pipeline: load -> chunk (500 chars, 50 overlap) -> embed each chunk -> save to document_chunks Result: ~7 rows in document_chunks (penalties, obligations, payment terms...) Chunks saved? Yes, permanently
The two pipelines process the same file independently

The extraction pipeline reads the file and saves 1 row in documents (with structured JSONB metadata). The RAG pipeline reads the same file and saves N rows in document_chunks (with embeddings). They link through document_id (foreign key). At query time, the extracted metadata is used for filtering (WHERE) and the chunk embeddings are used for searching (HNSW).

How the tables connect:

-- documents table (from extraction pipeline) id │ filename │ metadata (JSONB) ───┼───────────────────────┼───────────────────────────────── 2 │ contract_servicii.txt │ {"prestator": "CloudTech", "valoare": 45000} │ document_id = 2 (foreign key) │ -- document_chunks table (from RAG pipeline) id │ document_id │ chunk_index │ content │ embedding ───┼─────────────┼─────────────┼─────────────────────────────┼────────────── 1 │ 2 │ 0 │ "CONTRACT DE PRESTARI..." │ [0.02, -0.08, ...] 2 │ 2 │ 1 │ "VALOAREA CONTRACTULUI..." │ [0.06, 0.01, ...] 3 │ 2 │ 2 │ "OBLIGATIILE PRESTAT..." │ [-0.03, 0.05, ...] 4 │ 2 │ 3 │ "OBLIGATIILE BENEFIC..." │ [0.04, -0.01, ...]
RAG only uses the chunks for answering, not the extracted metadata

When RAG builds the context for the LLM prompt, it uses chunk.content (the text). The extracted metadata (numar, prestator, total) is never injected into the prompt -- it's only used for filtering with WHERE metadata->>'prestator' = '...' to narrow down which document's chunks to search.
