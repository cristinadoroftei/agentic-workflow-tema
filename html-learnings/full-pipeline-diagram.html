Two Chunkings, Two Purposes

↓ Chunking #1 — Lambda (Processing)
# Purpose: split big file for processing
# Size: larger (500-2000 chars)
# Destination: S3 (temporary JSON)
# Lifetime: temporary

chunks = chunk_text(body,
  size=500, overlap=50
)
These chunks are reassembled by stage 2 to get the full text, or the first 3 are selected for the LLM. They're a means to an end.
↓ Chunking #2 — RAG Indexing (Embeddings)
# Purpose: one chunk = one meaning = one vector
# Size: smaller (300-500 chars)
# Destination: document_chunks table (permanent)
# Lifetime: permanent

chunks = splitter.split(content)
for chunk in chunks:
  emb = model.encode(chunk)
These chunks are permanent. Each one becomes a row with an embedding vector. They're the searchable units for RAG.

Can we avoid chunking twice?

Yes, sometimes. If both stages use the same chunk size, stage 3 can reuse the S3 chunks directly instead of re-chunking. But often the sizes differ: processing chunks can be larger (the LLM handles 2000 chars fine), while embedding chunks should be smaller (one meaning per vector). So in practice, you chunk twice with different parameters.

Query Time — S3 Is Gone, Only PostgreSQL

At query time, S3 is never touched. Everything the app needs is in PostgreSQL across the two tables.
Q
User asks: "What are the penalty clauses in contracts?"
1
Filter by metadata
documents table
# Only search contracts, not invoices
docs = doc_repo.filter_by_metadata("doc_type", "contract")
doc_ids = [d.id for d in docs]
# doc_ids = [1, 3, 7] ← only contracts
This is why we extracted metadata. Without it, you'd search ALL document chunks (invoices, reports, everything) — noisy and slow.
doc_ids passed as filter
2
Semantic search within filtered chunks
document_chunks table
# Embed the user's question
query_emb = rag_service.embed("penalty clauses")

# Search ONLY in contract chunks (not all chunks)
results = chunk_repo.similarity_search(
  query_embedding=query_emb,
  doc_ids=doc_ids, # WHERE document_id IN (1, 3, 7)
  top_k=5
)

The LLM only sees relevant chunks, not entire documents. This is the whole point of RAG — precise context, better answers, fewer tokens.
