2. The Index = ONE Graph Over ALL Chunks

Each chunk is a node. Each edge means "these two chunks are close in vector space." The parameter m controls how many edges each node gets. This is the single HNSW index for the entire table.

Common misconception: "Each chunk gets its own small graph."
Reality: There is ONE graph. Each chunk is ONE node in that graph. When you insert a new chunk, it becomes a new node connected to its m nearest existing nodes. The graph grows, but it stays one unified structure.

3. Query Time: Searching the Graph

A user asks a question. It gets embedded into a vector, then the HNSW graph is traversed to find the nearest chunk nodes. The <=> operator computes cosine distance at each step.

4. HNSW vs IVFFlat

Two different ways to organize vectors for fast search. Same data, different index structures.

HNSW (graph-based)

Each vector = node. Edges connect similar vectors.
Search: traverse edges greedily toward query.
+ Fast queries (~2ms)
- High RAM (stores graph in memory)

IVFFlat (cluster-based)

Vectors grouped into clusters by centroids.
Search: find nearest centroid, scan that cluster.
+ Less RAM
- Slower, can miss results at cluster boundaries
