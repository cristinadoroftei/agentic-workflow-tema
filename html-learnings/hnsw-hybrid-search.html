How HNSW Behaves Differently

RAG Only: Search all chunks

The HNSW graph contains ALL chunks from ALL documents. The query vector enters at the top layer and greedily descends to layer 0, finding the nearest neighbors by cosine distance.

Problem: If you ask about TechDev's penalties, the top-5 closest chunks might be from other companies' contracts that also mention penalties. TechDev's specific clause might be ranked #7 and get cut off.

Hybrid: Filter THEN search

Before HNSW traversal, PostgreSQL applies WHERE document_id IN (3, 7). The graph still exists, but only nodes belonging to TechDev documents are considered as candidates.

Result: All top-5 results are from TechDev's documents. The penalty clause that was #7 globally is now #1 within the filtered set.

What PostgreSQL Actually Does

-- RAG only: searches entire HNSW graph SELECT chunk.*, 1 - (embedding <=> $query_emb) AS score FROM document_chunks chunk ORDER BY embedding <=> $query_emb LIMIT 5; -- Hybrid: filter first, HNSW only visits matching nodes SELECT chunk.*, 1 - (embedding <=> $query_emb) AS score FROM document_chunks chunk WHERE chunk.document_id IN (3, 7) -- TechDev's documents only ORDER BY embedding <=> $query_emb LIMIT 5;

Where does the WHERE come from?

The LLM doesn't write SQL. It calls a tool with parameters like { query: "penalitati", prestator: "TechDev" }. Your Python code runs filter_by_metadata("prestator", "TechDev") on the JSONB column to get doc_ids, then passes them into the similarity search.

How does the JSONB filter work?

The extraction pipeline (Lectia 3) saved structured data into documents.metadata as JSONB. PostgreSQL's ->> operator extracts a key from JSONB as text: WHERE metadata->>'prestator' = 'SC TechDev SRL'. This returns the document IDs to filter on.

Why not filter AFTER the HNSW search?

If you search globally with top_k=5 and then filter, you might end up with 0-2 results (the rest were from other companies). You can't just increase top_k to compensate -- you'd need to guess how many non-matching results there are. Filtering before the search guarantees exactly top_k results from the right documents.

Wrong: post-filter

1. HNSW returns top 5 globally
2. Filter: keep only TechDev
3. Result: maybe 1-2 chunks left
4. Penalty clause was #7 globally -- lost

Correct: pre-filter

1. WHERE limits candidates to TechDev
2. HNSW searches within those only
3. Result: exactly 5 TechDev chunks
4. Penalty clause is now #1 -- found

The Full Chain

User asks question --> LLM fills tool params --> JSONB filter (doc_ids) --> HNSW search (filtered) --> Top-k chunks --> LLM answers with context
