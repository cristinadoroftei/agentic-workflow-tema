Key principle: Don't force all data into one mechanism. SQL stays in SQL. NoSQL stays in NoSQL. RAG is for unstructured text only. Each agent speaks the native language of its data source.

How Does the LLM Know Which Source to Use?

You give it descriptions of each source: what data it holds, what schema it has, what the columns mean.

SQL
SQL Source

  • Type: PostgreSQL
  • Tables: customers, orders, invoices
  • customers: id, name, email, city
  • orders: id, customer_id, amount, date
  • "city" = orasul clientului

NoS
NoSQL Source

  • Type: MongoDB
  • Collections: locations, reviews
  • locations: address, city, geo_coords
  • reviews: customer_id, text, rating, date
  • "rating" = 1-5, sub 3 = negativ

CSV
CSV Source

  • Type: CSV files
  • Files: monthly_report.csv, sales.csv
  • Columns: region, revenue, month
  • "region" = cod judet/oras
  • Updated monthly

RAG
RAG Source

  • Type: pgvector + HNSW
  • Content: contracts, policies, legal docs
  • Metadata: doc_type, date, parties
  • Unstructured text, not queryable via SQL
  • Search by semantic similarity
This is what Richard meant by "expui sursele". The LLM sees these descriptions and can match a user's question to the right source. "Clienti din Bucuresti" → SQL (customers.city) or NoSQL (locations.city). "Review-uri negative" → NoSQL (reviews.rating). "Clauze de confidentialitate" → RAG (contract text).

How the LLM decides in practice

# The source descriptions become part of the system prompt

system_prompt = """You have access to these data sources:

1. SQL (PostgreSQL): customers, orders, invoices
   - customers: id, name, email, city
   - orders: id, customer_id, amount, date

2. NoSQL (MongoDB): locations, reviews
   - reviews: customer_id, text, rating (1-5), date

3. RAG: contracts, policies (unstructured text)

For each question, decide which source(s) to query.
You can query multiple sources if needed."""

RAG Is One Source, Not a Replacement for Everything

Richard's key point: don't force all data into RAG. Use the right mechanism for each data type.

Data Type Mechanism Why not RAG?
Relational data (customers, orders) SQL Already structured. SQL is precise — exact filters, joins, aggregations. RAG would lose that precision.
Flexible documents (reviews, profiles) NoSQL Already structured (JSON documents). You can query fields directly. No need for semantic search.
Tabular files (reports, exports) CSV / pandas Columns and rows. Filter and aggregate directly. RAG can't do SUM() or GROUP BY.
Unstructured text (contracts, policies) RAG This IS where RAG shines — text has no schema, can't be queried with SQL. Semantic search is the only option.
Richard's rule: "SQL-ul ramane in SQL. NoSQL-ul ramane in NoSQL, pentru ca deja este informatie structurata. Textul nestructurat il pui in RAG."

Multiple RAG Indexes for Different Document Types

If your documents are separable by type, Richard recommends a separate pgvector index for each:

contracts_chunks
Contract clauses, terms, parties, penalties
invoices_chunks
Invoice details, amounts, line items
medical_chunks
Medical documents, diagnoses, treatments
policies_chunks
Company policies, HR rules, guidelines
Why separate? Each index has its own HNSW graph. Searching contracts doesn't waste time comparing against medical documents. Better precision, faster search. Same principle as hybrid search — filter first, then search.

# LangGraph pattern (Lectia 5, slide 17)

def route_to_source(state):
  # LLM decides which branch based on the question
  source = llm.decide(state.question, available_sources)
  return source # "rag", "csv", or "sql"

graph.add_node("router", route_to_source)
graph.add_node("gen_rag_query", rag_handler)
graph.add_node("gen_csv_query", csv_handler)
graph.add_node("gen_sql_query", sql_handler)

# Conditional edges: router → one of the three branches
graph.add_conditional_edges("router", route_to_source, {
  "rag": "gen_rag_query",
  "csv": "gen_csv_query",
  "sql": "gen_sql_query",
})
Same architecture, different implementation. The multi-source pattern from Lectia 4 becomes concrete in LangGraph (Lectia 5) as nodes + conditional edges. The router node picks the branch, each branch generates a query in the right format.
